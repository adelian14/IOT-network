Performance Metrics Explained

How IDS Effectiveness Is Measured — IoT Network Security System

Source: metrics/tracker.py  ·  Companion document: offensive_module.html

Overview

Saying "the IDS detected the attack" is not a measurement — it is an anecdote. A rigorous evaluation requires knowing not only what the IDS caught, but also what it missed and what legitimate traffic it incorrectly flagged. This document explains exactly how the Metrics Tracker answers those questions.

The Metrics Tracker is a completely independent MQTT client. It has no access to the IDS's internal state, no knowledge of detection rules, and imports nothing from the security/ package. It evaluates the IDS purely as a black box, by comparing two external streams: what the offensive node says it attacked, and what the IDS says it detected.

The Three Channels

The Metrics Tracker subscribes to exactly three MQTT topics and records events from each independently:

home/#
Published by: sensors, CPU, attackers
Total message population. Every message on this wildcard is recorded with its topic and timestamp. This forms the complete universe of events that will be classified as TP, FP, TN, or FN.
metrics/ground_truth
Published by: offensive node only
Attack labels. Each label carries the attack type, the topic that was attacked, the payload, and a precise timestamp. This is the "true answer key". The IDS never sees this topic.
security/alert
Published by: IDS Decision Engine
IDS detections. Every SUSPICIOUS or THREAT verdict produces an alert on this topic. The tracker records the alerted topic and its timestamp. These are the IDS's "predictions".

The Correlation Algorithm

Attack messages and IDS alerts do not arrive at exactly the same millisecond. There is latency through the broker, IDS processing time, and the LLM call (if invoked). To account for this, the tracker uses a ±2-second time window to match events to messages.

Step-by-Step

At shutdown, the tracker runs compute_metrics(), which does the following:

  1. Builds a lookup table: topic → [sorted list of ground truth timestamps]
  2. Builds a lookup table: topic → [sorted list of IDS alert timestamps]
  3. Iterates over every message seen on home/#. For each message m with topic T and timestamp t:
    • is_attack = True if any ground truth timestamp for topic T falls within t ± 2 s
    • is_detected = True if any IDS alert timestamp for topic T falls within t ± 2 s
  4. Classifies the message as TP, FP, TN, or FN based on those two booleans.
Why ±2 seconds?
The IDS runs asynchronously: the LLM Analyzer (Gemini) is invoked in a background thread and may take 0.5–1.5 seconds to return. A 2-second window comfortably absorbs this latency without being so wide that it creates false correlations on busy topics.

Metric Definitions

Accuracy

The fraction of all messages that the IDS classified correctly — both attacks it correctly flagged and legitimate traffic it correctly passed through.

Accuracy = TP + TN TP + TN + FP + FN
Numerator All correct decisions — attacks detected (TP) plus normal traffic passed (TN) Denominator Every message ever seen — the complete observation set Range 0.0 (every message misclassified) to 1.0 (perfect classification) Limitation Can be misleadingly high if attacks are rare (e.g., 5 attacks in 1000 messages — ignoring all attacks still gives 99.5% accuracy)
Quick example
TP=35, TN=157, FP=3, FN=5 → Accuracy = (35+157) / (35+157+3+5) = 192/200 = 96.0%

Precision

Of all the messages the IDS raised an alert on, what fraction were genuine attacks? High precision means when the IDS fires, it is usually right — few false alarms.

Precision = TP TP + FP
Numerator Alerts that were correct (real attacks) Denominator All alerts fired — correct ones plus false alarms Range 0.0 (every alert is a false alarm) to 1.0 (every alert is a real attack) Practical meaning Low precision → operator alert fatigue; too many false alarms cause legitimate alerts to be ignored
Quick example
TP=35, FP=3 → Precision = 35 / (35+3) = 35/38 ≈ 92.1%
The IDS raised 38 alerts total. 35 were real attacks, 3 were false alarms.

Recall (Sensitivity / Detection Rate)

Of all the actual attacks that occurred, what fraction did the IDS successfully detect? High recall means few attacks slip through undetected.

Recall = TP TP + FN
Numerator Attacks that were detected Denominator All attacks that actually occurred — detected ones plus those that were missed Range 0.0 (no attacks detected) to 1.0 (every attack caught) Practical meaning Low recall → attackers succeed undetected; in security contexts, recall (catching attacks) is often more important than precision
Quick example
TP=35, FN=5 → Recall = 35 / (35+5) = 35/40 = 87.5%
40 attack messages occurred. The IDS caught 35 of them, missed 5.

F1 Score

The harmonic mean of Precision and Recall. It captures both concerns in a single number — useful when you want a balanced view of an IDS that must minimise both false alarms and missed detections simultaneously.

F1 = 2 × Precision × Recall Precision + Recall

This is equivalent to the harmonic mean of the two values. It penalises extreme imbalances more severely than the arithmetic mean would. For example, Precision = 1.0 and Recall = 0.5 gives an arithmetic average of 0.75 but an F1 of only 0.667 — correctly reflecting that missing half the attacks is a serious problem even if every alert fired is accurate.

The F1 can also be expressed directly in terms of the confusion matrix:

F1 = 2 × TP 2 × TP + FP + FN
Range 0.0 (worst) to 1.0 (perfect precision and recall) Practical meaning F1 is the go-to summary metric for IDS evaluation; it degrades sharply if either FP or FN is high
Quick example
Precision=0.921, Recall=0.875 → F1 = 2 × (0.921 × 0.875) / (0.921 + 0.875)
                                 = 2 × 0.8059 / 1.796 = 1.612 / 1.796 ≈ 89.7%

The Precision–Recall Trade-off

Precision and Recall naturally pull against each other. An IDS can be tuned to detect more attacks (higher Recall) by lowering its detection threshold — but this also increases false alarms (lower Precision). Conversely, a very conservative IDS raises fewer alarms (higher Precision) but misses more attacks (lower Recall).

Per-Attack Detection Rate

In addition to the overall confusion matrix, the tracker computes a per-attack-type detection rate. This uses the metrics/ground_truth events directly (not the home/# messages) as the unit of measurement:

Detection Ratetype = attacks of type T detected by IDS attacks of type T sent by attacker

For each ground truth event of a given type, the tracker checks whether the IDS fired an alert on the same topic within ±2 seconds. This gives a direct answer to questions like "what percentage of injection attacks did the IDS catch?"

Unit Difference
The confusion matrix counts home/# messages as its unit. The per-attack detection rate counts ground truth events as its unit. These are different: one attack event may correspond to multiple messages in a flooding scenario. Both views are useful; they answer different questions.

Step 3 — Per-Attack Detection Rate

The tracker separately iterates over all 40 ground truth events, grouped by attack type, and checks whether an IDS alert fired for that topic within ±2 s:

Attack Type GT Events Sent Detected by IDS Detection Rate Primary Rule Triggered
spoofing 10 10 100.0% value_range sev 8
flooding 12 12 100.0% Firewall rate limit
injection 8 8 100.0% direct_actuator_access sev 9
replay 6 4 66.7% Rate limit · suspicious_pattern
malformed 4 1 25.0% payload_integrity sev 4–7

Injection and spoofing are detected at 100% because their primary rules assign severity ≥ 7 immediately, bypassing any threshold ambiguity. Replay is harder because replayed payloads look legitimate — only the frequency is abnormal, and the rate-limit window may not have filled yet on every message. Malformed payloads are the hardest: some patterns only reach severity 4 (SUSPICIOUS, not THREAT), so they don't always reach the security/alert channel that the Metrics Tracker watches.

Reading the Generated HTML Report

At shutdown (Ctrl+C), the tracker calls compute_metrics(), prints results to stdout, and saves an HTML report to reports_output/metrics_YYYYMMDD_HHMMSS.html. The report contains:

  • Session totals — message count, ground truth events, alerts fired
  • Confusion matrix — colour-coded TP/FP/TN/FN grid
  • Metric cards — Accuracy, Precision, Recall, F1 with colour scale (green ≥ 90%, amber ≥ 70%, red below)
  • Per-attack breakdown table — detection rate for each attack type
Interpreting the Results

A healthy IDS in this system should show:

  • Accuracy > 95% — most messages are legitimate and correctly left alone
  • Precision > 85% — alerts are trustworthy, low false alarm rate
  • Recall > 80% — most attacks are caught; some replay/malformed may slip below threshold
  • F1 > 85% — a balanced indicator of overall detection quality
