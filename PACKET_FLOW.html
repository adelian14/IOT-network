Section 3 — Step 1

A Message Arrives: Building the PacketRecord

When any MQTT message comes in, IDSMonitor._on_message() fires. The raw message only contains a topic and a binary payload. The IDS enriches it into a structured PacketRecord object by cross-referencing config files and system state.

Data Engineering at This Stage

Each raw field goes through a deliberate transformation before being stored in the record:

topic
Used directly
"home/temp/data"
payload
Decoded bytes → string
"23.5"
client_id
topic → mqtt_topics.json → network_config.json → "{name}-sensor"
"temp-sensor"
source_ip
client_id → network_config.json lookup
"10.0.0.3"
timestamp
Current system time added
"2026-03-31T14:22:05.123"
💡 Why enrich the packet? Raw MQTT does not include sender identity or IP — those are inferred from the topic and config. This enrichment step is what makes every downstream stage possible.

The Resulting Python Object

Python
PacketRecord( topic = "home/temp/data", payload = "23.5", client_id = "temp-sensor", source_ip = "10.0.0.3", timestamp = "2026-03-31T14:22:05.123456" )

This object is passed to every downstream stage and is serializable via .to_dict() — which is the exact form sent to Gemini. The IDS also keeps a rolling history of the last 100 messages per device in _device_history, used later for context.

Section 4 — Step 2

The Firewall

Before any analysis begins, the firewall runs six fast checks. Its job is to reject obvious bad actors and flag suspicious-but-uncertain ones.

#CheckDescription
1IP BlacklistIs source_ip in the static block list?
2IP WhitelistIs source_ip in the allowed list at all?
3Dynamic BlockWas this device temporarily blocked by a prior self-healing action?
4Rate LimitHas this client_id sent too many messages in the time window?
5Client ID CheckIs this a known device?
6Payload SizeIs the payload too large?
PASS All clear

All checks passed. Continue to Rule Engine.

FLAG Suspicious

Odd but not blocked. Reasons are injected as fake rule results downstream.

BLOCK Hard stop

Packet dropped. A severity-9 threat is created immediately. All further stages are skipped.

⚠️ FLAG is clever: when a packet is flagged, the firewall converts its reasons into fake RuleResult objects and injects them into the rule engine output. This means FLAG reasons are treated the same as any other rule trigger downstream — no special-casing needed.
Section 5 — Step 3

The Rule Engine

The rule engine runs six independent checks, each returning a severity (0–10) and a confidence (0.0–1.0). These two numbers drive every decision that follows.

value_range
Sensor reading within physical bounds?

Checks if the payload value is within the expected physical range for that sensor type.

⚡ Temperature sensor sends 9999
topic_acl
Sender authorized for this topic?

Verifies that the client_id is on the allowed-publishers list for this specific MQTT topic.

⚡ Sensor publishes to an actuator control topic
payload_integrity
Payload contains attack strings?

Scans for binary data, SQL injection, XSS, and path traversal patterns.

⚡ Payload contains "SELECT * FROM"
direct_actuator_access
Non-CPU sending commands to actuator?

Only the central CPU should ever publish to actuator control topics.

⚡ temp-sensor publishes to home/thermo/control
unknown_topic
Topic not in the registry?

Any message on an unregistered topic is inherently suspicious — legitimate devices don't invent new topics.

⚡ Message on home/hacked/data
suspicious_patterns
Payload unusually large or repetitive?

Detects flooding (very large payloads) and DoS patterns (repeated identical bytes).

⚡ 600-char payload or "AAAAAAAAAA" repeated

What the Rule Engine Outputs

Python tuple
( triggered_rules, # list of dicts — one per triggered rule severity, # highest severity among triggered rules (int 0–10) confidence, # lowest confidence among triggered rules (float 0.0–1.0) needs_llm # True if severity >= 5 AND confidence <= 0.6 )

Example: High-Confidence Detection (Gemini NOT called)

Spoofing — value out of range
Output
triggered_rules = [{ "rule_name": "value_range", "triggered": True, "severity": 9, "confidence": 0.85, "detail": "Value 9999.0 outside range [-50, 150] for temp" }] severity = 9 confidence = 0.85 needs_llm = False # confidence > 0.6 → Gemini NOT called

Example: Ambiguous Detection (Gemini IS called)

Unknown topic — low confidence
Output
triggered_rules = [{ "rule_name": "unknown_topic", "triggered": True, "severity": 5, "confidence": 0.5, "detail": "Message on unregistered topic: 'home/mystery/data'" }] severity = 5 confidence = 0.5 needs_llm = True # severity >= 5 AND confidence <= 0.6 → Gemini called

The Escalation Condition — When Does Gemini Get Called?

needs_llm = (severity ≥ 5) AND (confidence ≤ 0.6)

Think of it this way: the rule engine is confident when confidence is high (close to 1.0). Gemini is only called when the rules are uncertain — i.e., when confidence is low. High-confidence detections like a sensor sending 9999 are dealt with directly, no Gemini needed.

Section 6 — Step 4

Gemini LLM Analysis

Gemini is the intelligence layer that handles ambiguous cases the rule engine can't confidently resolve. It runs asynchronously — in a background thread — so it never blocks the main monitoring loop.

What Data Is Passed to Gemini?

Section 7 — Step 5

The Decision Engine

The decision engine combines everything collected so far into a single final verdict. Gemini's output can move the severity number in both directions.

How Gemini's Output Modifies Severity

Gemini SaysEffect on Severity
"is_threat": false Severity is reduced by 3 — Gemini cleared it
"is_threat": true Severity is bumped up to at least the Gemini level critical→10 / high→8 / medium→6 / low→3

Confidence is also updated: it takes the maximum of the rule confidence and Gemini's confidence.

Verdict Thresholds

≥ 7 THREAT

Self-heal, alert, and generate HTML report.

4 – 6 SUSPICIOUS

Logged and monitored. No immediate action.

< 4 SAFE

No action taken.

What Happens on THREAT?

1
Create ThreatEvent

A permanent record of the incident is created and appended to incident_log.

2
Publish Self-Healing Action

Built from llm_result["recommended_actions"] (or falls back to rule-based logic) and published to security/action as JSON for the CPU to execute.

3
Publish Alert

An alert is published to the security/alert topic.

4
Generate HTML Incident Report

An HTML report is written immediately to reports_output/.

Section 10

Key Design Choices

These decisions shape the entire system's behavior and tradeoffs:

⚡
Gemini is non-blocking

It runs in a background asyncio event loop. The main MQTT listener never waits for it — high-throughput environments won't bottleneck at the LLM call.

🎯
Gemini handles only ambiguous cases

High-confidence detections (like confidence=0.95 on direct actuator access) skip Gemini entirely. LLM calls are reserved for genuinely uncertain situations.

✅
Gemini can clear a threat

If "is_threat": false comes back, the severity drops by 3 — potentially moving the verdict from THREAT to SUSPICIOUS or even SAFE. The LLM is a two-way gate.

🔧
LLM chooses the corrective action

Gemini's recommended_actions are used directly as the self-healing command to the CPU. The model effectively decides what happens to the offending device.

🕐
Device history provides critical context

The last 10 messages from each device are included in the Gemini prompt so it can spot sudden behavioral changes — e.g., three normal lux readings followed by a SQL string.

🔌
Graceful fallback when Gemini is unavailable

If the API key is missing or the call fails, _fallback_analysis() produces an identical JSON structure from rule results, with "source": "fallback".

🏁 That's the complete pipeline. Every MQTT message that passes through this system is classified from a raw topic + binary payload into one of three verdicts — SAFE, SUSPICIOUS, or THREAT — with a full audit trail, optional LLM analysis, and automated self-healing when needed.
