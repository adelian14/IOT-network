Offensive Module

Attack Techniques & Packet Flow — IoT Network Security System

Source: offensive/  ·  Node: 10.0.0.50  ·  Companion document: metrics_explained.html

Overview

The Offensive Node is the red-team component of the IoT Network Security System. It simulates a malicious actor inside the 10.0.0.0/24 subnet who has gained network access and is attempting to compromise the smart-home infrastructure.

Its goals are straightforward: send realistic attacks across the MQTT broker, observe how the IDS responds, and provide objective ground-truth labels that allow the Metrics Tracker to evaluate detection accuracy without trusting the IDS's own self-assessment.

The module is deliberately not stealthy — it uses recognisably fake client IDs and out-of-range payloads so that each attack triggers well-defined detection rules. This makes it a controlled, repeatable test harness rather than an actual evasion tool.

Position in the Network

Every device communicates exclusively through the MQTT broker at 10.0.0.100. The Attacker connects to the same broker, making it invisible from a routing perspective — it looks like just another MQTT client. The IDS, however, watches every topic on the broker and classifies each message through a four-stage pipeline.

The Dual-Publish Mechanism

Every MQTT attack module follows the same two-action pattern. For each malicious message it publishes, it immediately calls ground_truth.tag() to publish a parallel label to the metrics/ground_truth topic. These two publications are always paired.

Why two separate publications?
The IDS must evaluate traffic on its own merits — it cannot know in advance which messages are attacks. By putting ground truth labels on a separate topic that the IDS explicitly ignores, the system can measure real detection performance without any information leaking into the IDS pipeline.

The Six Attacks

1 Network Scan — Reconnaissance scanner.py

The scan is a pre-attack reconnaissance phase. It discovers which hosts are alive on the 10.0.0.x subnet and identifies the MQTT broker before any MQTT messages are sent. This mirrors what a real attacker would do before targeting specific devices.

How it works

  1. Opens a raw TCP socket to each IP in the range 10.0.0.1 – 10.0.0.100 on port 1883 (MQTT default).
  2. Records any IP where the TCP handshake succeeds — these hosts have Mosquitto running.
  3. For each discovered host, sends a raw MQTT CONNECT packet (byte-crafted, no library) and reads the CONNACK return code to confirm broker identity.
  4. Scans discovered hosts for common additional ports (22 SSH, 80 HTTP, 443 HTTPS, 8080, 8883 MQTTS).
Protocol levelTCP (not MQTT) — no broker authentication MQTT messages sentNone — only raw socket connections Ground truth tagsNone — no MQTT, so no tagging IDS visibilityNone — the IDS only sees MQTT traffic via the broker
Detection Gap
The network scan produces zero MQTT messages and is therefore completely invisible to the IDS. This is intentional — it demonstrates the IDS's blind spot at the network/transport layer. A real deployment would pair this with network-level IDS (Snort/Suricata) or firewall logs.
2 Spoofing — Fake Sensor Data attacks/spoofing.py

The attacker impersonates IoT sensors by publishing to their data topics with physically impossible values. The goal is to feed the CPU with false readings that trigger incorrect actuator commands — for example, reporting a temperature of 999°C to force the thermostat to maximum cooling.

How it works

  1. Connects to the broker using client ID spoofed-sensor (not in the device registry).
  2. Every 1 second, picks a random target topic and a random out-of-range value from a preset table.
  3. Publishes the fake payload to that topic.
  4. Immediately calls gt_tag() to publish the label to metrics/ground_truth.

3 Flooding — Denial of Service attacks/flooding.py

The attacker floods the broker with a high volume of messages to overwhelm the CPU's message processing loop and degrade the IDS's ability to keep up with traffic. Random payloads are used so individual messages don't trigger value-range rules; the volume itself is the attack vector.

How it works

  1. Connects to the broker using client ID flood-bot.
  2. Sends messages at a configurable rate (default: 50 messages/second).
  3. Cycles through a mix of real device topics (home/*/data) and fake topics (flood/random/topic1) to probe different detection rules.
  4. Each message payload is a random alphanumeric string (10–200 chars) to avoid pattern detection.
  5. Tags every message with a ground truth label.
Client IDflood-bot Rate50 msgs/sec (default) for 15 s → ~750 messages total Primary detectionFirewall rate limit — sensor type allows ≤ 15 msgs/60 s; 50/s exceeds this immediately Secondary triggersunknown_topic (severity 5) for flood/* topics · unknown client ID
Rate Limit Detail
The Firewall uses a sliding-window counter per device type. Sensors are allowed 15 messages per 60-second window. Once exceeded, the message is flagged and the severity escalates to the point of a THREAT verdict and dynamic block.
4 Injection — Direct Actuator Commands attacks/injection.py

This is the highest-severity attack in the system. The attacker bypasses the CPU entirely and publishes commands directly to /control topics — the channel reserved exclusively for the CPU node. This can physically manipulate actuators: unlock doors, disable cameras, silence fire alarms, or create electrical hazards.

How it works

  1. Connects as client ID injector-node.
  2. Cycles through a list of pre-defined (topic, command) pairs at 2-second intervals.
  3. Each pair targets a specific actuator control topic and sends a dangerous command.
  4. Tags each command with a ground truth label.

Highest Severity
The direct_actuator_access rule assigns severity 9 (the maximum) to any non-CPU client that writes to a /control topic. This immediately triggers a THREAT verdict without needing LLM confirmation.
5 Replay — Capture & Replay attacks/replay.py

A two-phase attack that first passively captures legitimate sensor messages, then floods the broker with them at an abnormally high rate. Because the payloads are valid (captured from real sensors), they don't fail value-range checks — the anomaly is the frequency and repetition.

How it works

  1. Capture phase: Connects as replay-sniffer, subscribes to five sensor topics, and passively records all messages for 10 seconds. No ground truth labels are published during this phase — sniffing legitimate traffic is not itself an attack event.
  2. Replay phase: Disconnects, connects as replay-attacker, and immediately replays the captured messages in a tight loop at 5 messages/second.
  3. Each replayed message is tagged with a ground truth label (attack type: replay).
Client IDsreplay-sniffer (capture) · replay-attacker (replay) Capture topicshome/thermo/data, home/lighting/data, home/door/data, home/camera/data, home/sound/data Replay rate5 msgs/sec for 20 s → ~100 replayed messages Primary detectionFirewall rate limit exceeded · suspicious_pattern (repeated identical payloads)
Intentional Asymmetry in Ground Truth
Only the replay phase is tagged. The capture phase is passive observation of legitimate traffic and would be impossible to distinguish from a benign subscriber (such as the Metrics Tracker itself). Tagging it would create false positives in the ground truth data.
6 Malformed Payloads — Input Validation Attack attacks/malformed.py

Sends deliberately corrupt or malicious payloads to test the IDS's input validation and the CPU's parsing robustness. Payloads range from binary garbage and SQL injection strings to oversized buffers and prototype pollution attempts.

How it works

  1. Connects as malform-bot.
  2. Every 1.5 seconds, picks a random target topic and a random payload from a curated list of malicious strings.
  3. Publishes the payload and tags it with ground truth (payload truncated to 100 chars for the label).

Packet Flow Through the IDS Pipeline

Once an attack message arrives at the broker, the IDS (subscribed to #) receives it and immediately passes it through four sequential stages. Here is what happens to a spoofed temperature reading of 999:

Ground Truth System

The file offensive/ground_truth.py exposes a single function, tag(), that every attack module calls immediately after publishing each malicious message. 

# offensive/ground_truth.py

def tag(client, attack_type: str, topic: str, payload: str) -> None:
    event = {
        "attack_type": attack_type,   # e.g. "spoofing", "flooding"
        "target_topic": topic,         # the topic the attack was sent to
        "payload": str(payload)[:200], # truncated for safety
        "timestamp": datetime.now().isoformat(),
    }
    client.publish("metrics/ground_truth", json.dumps(event))

The Metrics Tracker subscribes to metrics/ground_truth and stores every event in a list. At shutdown it uses these timestamps to determine — for each message on home/# — whether the message was an attack or legitimate traffic (see metrics_explained.html).

The No-Leakage Guarantee

A critical requirement of any fair IDS evaluation is that the IDS must not be able to read the ground truth labels. If it could, a naive implementation might cheat by treating the presence of a metrics/ground_truth event as a detection signal. This system prevents that with two independent mechanisms:

Mechanism 1 — Topic filter at the IDS entry point.
The very first lines of _on_message() in security/ids_monitor.py discard any message whose topic starts with security/ or metrics/. The message is dropped before the Firewall, Rule Engine, LLM, or Decision Engine are even instantiated for that message. There is no code path from metrics/ground_truth to any IDS stage.
Mechanism 2 — Zero import coupling.
metrics/tracker.py does not import any module from security/. offensive/ground_truth.py does not import any module from security/. The dependency graph is strictly one-directional: both the offensive node and the metrics tracker are peers of the IDS, not extensions of it.

Running the Offensive Node

The offensive node supports three modes:

# Interactive menu — choose attacks one at a time
python offensive/attacker.py

# Automated sequence — all six attacks in order with 5 s pauses
python offensive/attacker.py auto

# Network recon only
python offensive/attacker.py scan

# Single attack by number (1–6)
python offensive/attacker.py 2   # spoofing only

In automated mode the attack sequence runs with these default durations:

Recommended Test Order
Start the Metrics Tracker (python -m metrics.tracker) before launching the offensive node so it catches all ground truth labels. Run the IDS before that. Stop with Ctrl+C in reverse order: offensive node first, then IDS (generates summary report), then Metrics Tracker (generates metrics report).
