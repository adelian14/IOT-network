Gemini is not always called. If the rule engine is already highly confident (e.g. a sensor reports a physically impossible value), the system handles it directly without touching the LLM. Gemini is reserved for genuinely ambiguous situations.
Trigger Condition

When Does Gemini Get Called?

The rule engine outputs two numbers for each packet: a severity (how bad) and a confidence (how sure the rules are). Gemini is only summoned when both conditions are true simultaneously:

⚡
severity ≥ 5 AND confidence ≤ 0.6

Translation: "Something looks bad, but we're not sure enough to act on our own." High-confidence detections skip Gemini entirely.

Input

What Gemini Receives

Three pieces of context are bundled into every Gemini prompt — the current packet, the rule results, and a short history of the device's recent messages:

📦 The Packet

Topic, payload, device ID, source IP, and timestamp of the flagged message.

📋 Rule Results

Which rules fired, with their severity and confidence scores, and a plain-language detail string.

📜 Device History

The last 10 messages from this same device — so Gemini can spot sudden behavioral changes.

The prompt also describes the full network context — 10 sensors, 10 actuators, a central CPU, and the expected MQTT topic structure — so Gemini can reason about whether behavior is normal for this type of device.

Gemini Prompt Structure (simplified)
You are an IoT network security analyst for a smart-home network. ## Flagged Packet - Topic: home/light/data - Payload: SELECT * FROM users - Device: light-sensor (10.0.0.2) - Time: 2026-03-31T14:35:12 ## Rules That Triggered payload_integrity: SQL injection pattern found (severity=7, confidence=0.6) ## Recent History from this Device [14:35:10] home/light/data → 1023 [14:35:11] home/light/data → 1024 [14:35:12] home/light/data → SELECT * FROM users ← flagged ## Network Context 10 sensors, 10 actuators, central CPU at 10.0.0.100 Expected light sensor payloads: numeric lux values ## Required Response Respond ONLY with valid JSON: { "threat_level": "none|low|medium|high|critical", "threat_type": "spoofing|injection|dos|unauthorized...", "explanation": "your full reasoning", "is_threat": true or false, "recommended_actions": ["action1", "action2"], "confidence": 0.0 to 1.0 }
Output

What Gemini Returns

Gemini responds with plain JSON — no markdown, no prose preamble. The IDS parses this directly. Here's what a real response looks like for the SQL injection example:

🤖 Gemini Response — SQL Injection Case
{ "threat_level": "high", "threat_type": "injection", "is_threat": true, "confidence": 0.92, "explanation": "The payload 'SELECT * FROM users' is completely out of place for a light sensor, which should only publish numeric lux values (e.g. 1023). The device history confirms normal readings just before this message — this was almost certainly crafted by an attacker probing for SQL injection.", "recommended_actions": ["block_device", "increase_monitoring"] }

Notice the explanation field — Gemini uses the device history to contrast the anomalous payload against the normal pattern of numeric lux readings. This is contextual reasoning that the rule engine alone cannot perform.

Decision Power

What Gemini Actually Controls

Gemini's response feeds directly into the Decision Engine and influences three things:

If is_threat = true
Severity is raised

The final severity is bumped up to match Gemini's threat level — critical → 10, high → 8, medium → 6, low → 3. Confidence is updated to whichever is higher.

If is_threat = false
Severity drops by 3

Gemini can clear a threat. A packet that triggered rules with severity 7 might drop to 4 (SUSPICIOUS) or even lower if Gemini sees a benign explanation.

recommended_actions
Gemini chooses the corrective action

recommended_actions from Gemini's JSON is used directly as the self-healing command sent to the CPU via security/action. The model effectively decides what happens to the offending device — block it, increase monitoring, isolate it, etc.

Final Verdict Thresholds

After Gemini's adjustments, the Decision Engine maps the final severity number to one of three outcomes:

≥ 7 THREAT

Self-heal, publish alert, generate HTML incident report.

4 – 6 SUSPICIOUS

Logged and monitored. No immediate action.

< 4 SAFE

No action taken.

After a THREAT Verdict

What Happens When Gemini Confirms a Threat

When the final verdict is THREAT, four things fire in sequence — and Gemini's output shapes all of them:

  • 01
    Incident record created. A permanent ThreatEvent is logged with the full packet, rule results, Gemini's JSON, and the final verdict.
  • 02
    Self-healing action published to security/action. Built from Gemini's recommended_actions — e.g. { action: "block_device", device: "light-sensor", duration: 300 } — sent to the CPU to execute.
  • 03
    Alert published to security/alert. A short alert message is broadcast on the network.
  • 04
    HTML incident report saved. A human-readable report is written to reports_output/, containing Gemini's full explanation paragraph verbatim alongside all packet and rule details.
Resilience

What If Gemini Is Unavailable?

🔌
Graceful fallback — the system never stalls.

If the Gemini API key is missing or the call fails, _fallback_analysis() generates an identical JSON structure mechanically from the rule engine results. Confidence is capped at 0.6 and the source is marked "fallback" so the report makes it clear LLM analysis was unavailable. Every downstream stage proceeds exactly the same.

Bottom line: Gemini is a powerful but optional intelligence layer. The system is fully functional without it — it just loses the contextual reasoning, behavioral history analysis, and natural-language explanations that the LLM uniquely provides.
